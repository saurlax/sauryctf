# 比赛参与闭环

当前实现覆盖了最基础的一段比赛体验：选手先加入队伍，再报名比赛，随后以当前队伍视角查看题目和提交 Flag。

## 当前能力

- 公开比赛详情页现在支持未登录访客直接浏览
  - 可以查看比赛基础信息、比赛状态、公开排行榜
  - 可以查看题目标题、分类、分值、解题数
  - 不能直接看到完整 `description` / `hints` / `attachments`
- 新增 `GET /api/games/{id}/participation`
  - 返回当前用户是否有队伍
  - 返回当前队伍是否已报名该比赛
  - 返回当前参与状态与队伍摘要
  - 如果管理员已分配分组，也会返回当前队伍的 `division`
  - 也会返回 Writeup 相关摘要：
    - 当前比赛是否要求 Writeup
    - Writeup 截止时间
    - 当前队伍是否已提交
    - 截止后是否仍处于未提交状态
- 比赛报名默认进入 `pending`
  - 需要管理员在管理端审核为 `accepted`
  - `rejected` 状态下队伍可以根据提示重新提交报名
  - 如果比赛使用 `auto_accept`，报名后会直接进入 `accepted`
  - 如果比赛设置了 `max_team_members`，超过人数上限的队伍不能报名
- `GET /api/games/{id}/challenges`
  - 未登录访客也可以读取公开比赛的题目基础信息
  - 如果当前用户所属队伍已报名，会返回队伍视角的题目状态
  - 包括 `solved` 以及一血 / 二血 / 三血队伍
  - 未通过报名、未开始前的已通过队伍、以及普通查看者，默认只能看到题目基础信息
  - 完整 `description` / `hints` / `attachments` 会在“已通过报名 + 比赛开始后”开放
- `GET /api/games/{id}/scoreboard`
  - 会包含已报名但暂未出分的队伍
- `GET /api/games/{id}`
  - 会返回比赛当前的 `registration_mode`
  - 会返回比赛当前的 `max_team_members`
  - 会返回比赛当前的 `divisions`
  - 也会返回 `practice_mode`、`writeup_required`、`writeup_deadline`
- 选手侧 Writeup 读写接口现在也进入了 OpenAPI 契约：
  - `GET /api/games/{id}/writeup`
  - `PUT /api/games/{id}/writeup`

## 前端行为

- 比赛详情页会展示：
  - 当前报名状态
  - 一个“当前下一步”提示，直接告诉选手现在最推荐的操作
  - 一个“当前报名会发生什么”提示：
    - 未登录时先提示需要登录
    - 未组队时先提示需要准备队伍
    - 未报名时会明确区分三种报名路径：
      - 邀请码 + 自动通过
      - 邀请码 + 待审核
      - 无邀请码 + 待审核 / 自动通过
    - 已报名后会明确区分待审核、已拒绝、已通过
  - 一张更聚焦的“当前参赛路径”卡片：
    - 会把登录、准备队伍、完成报名、开始解题这几步按当前状态重新组织
    - 会根据“未登录 / 没队伍 / 未报名 / 待审核 / 已通过”切换提示文案
  - 一个“参赛步骤”摘要，明确区分登录、组队、报名、开始解题四个阶段
  - 一组更紧凑的比赛信息摘要，集中展示报名方式、分组、人数限制、赛后练习和 Writeup 要求
  - 当前队伍名称
  - 当前比赛分组列表
  - 当前队伍所属分组
  - 管理员填写的比赛公告
  - 比赛是否支持赛后练习
  - 比赛是否要求 Writeup 以及截止时间
  - 当前队伍的 Writeup 提交入口与审核状态
  - 待审核 / 已拒绝 / 已通过的区别说明
  - 报名比赛 / 退出比赛操作
  - 当前是否可报名 / 可退赛 / 可提交 Flag 的原因说明
  - 如果比赛不需要邀请码，报名区不会再显示一个空的邀请码输入框，而是直接说明“这次报名会自动通过”或“这次报名会进入待审核”
- 每道题目的题面、提示列表和附件入口
- 如果题目填写了 `container_spec`：
- 当前比赛页会把它作为“实例接入信息”展示
  - 适合放实例 URL、靶机 host/port、连接命令或代理入口说明
  - 现在也支持在同一结构里附带 `runtime.provider`、`runtime.image`、`runtime.expose`
  - 适合先沉淀“这道题应该跑在哪、暴露哪些端口”，为后续真正接动态容器预留稳定结构
  - 如果接入信息里使用了 `{{team_hash}}` 这类占位符：
    - 启动实例前，比赛页会明确提示当前显示的是入口模板
    - 启动后会优先展示后端已经为当前队伍解析好的真实租约入口
  - 对于 `dynamic` 题目，如果同时提供了 `runtime.provider` 或 `runtime.image`：
    - 当前比赛页还会额外展示一个“实例状态”区
    - 已通过报名的队伍可以主动启动实例，或在租约快结束时继续续期
    - 默认实现仍然是最小实例租约骨架，只负责记录运行状态、到期时间和入口信息
    - 当后端显式开启 `INSTANCE_DOCKER_PROVIDER_ENABLED=true` 后，`runtime.provider = docker` 会切换到本地真实 Docker CLI provider
    - 当前这个真实 provider 适合先跑最小 Web 容器链路，例如 `nginx:alpine + expose: [80]`
  - 在未满足题面可见条件时，这部分会和题面、提示、附件一起隐藏
- 按分类分组的题目列表
- 当题面尚未开放时，前端会明确提示当前仍处于隐藏状态，而不是直接显示空白
- 比赛详情页的“当前下一步”会覆盖这些常见状态：
  - 未登录
  - 未加入队伍
  - 尚未报名
  - 报名待审核 / 已拒绝
  - 已报名待开赛
  - 已开始可直接解题
  - 待补交 Writeup / Writeup 待审核
- 如果游客是从某个比赛详情页进入登录或注册：
  - 页面现在会保留当前比赛地址到 `redirect`
  - 登录或注册成功后会直接回到原比赛页，继续报名或查看题目
  - 登录页和注册页之间互相切换时，也会继续保留这个 `redirect`
- 如果已登录用户是从某个比赛详情页进入队伍页：
  - 比赛页里的“去队伍页 / 去加入队伍 / 查看队伍 / 管理我的队伍”会保留当前比赛 `redirect`
  - 创建队伍或加入队伍成功后会自动回到原比赛页
  - 队长复制邀请链接时，也会把这个比赛返回地址一起带给队友
- 比赛概览页当前会优先按这三个信息块组织：
  - 比赛简介 / 公告 / 参赛须知
  - 当前行动与推荐入口
  - 当前报名与比赛配置信息
- 如果比赛启用了分组，概览页会额外解释当前分组规则：
  - 未启用分组：直接进入统一榜单
  - 只配置一个分组：报名成功后自动归入该分组
  - 配置多个分组：报名后先建立参赛记录，后续由管理员分配具体分组
- 题目区现在会直接解释为什么当前不能提交 Flag，而不是只禁用输入框
- Writeup 标签也会明确区分这些状态：
  - 当前比赛不要求 Writeup
  - 尚未组队 / 尚未报名 / 报名待审核
  - 已过 Writeup 截止时间
  - 可以重新提交，或被驳回后重新提交
- Writeup 标签页当前会优先按这三个信息块组织：
  - 状态摘要与是否可提交
  - 当前审核信息与备注
  - 提交流程说明与正文编辑区
- 比赛列表页也会展示当前用户的参赛入口状态：
  - 未登录
  - 未加入队伍
  - 可报名
  - 已报名
  - 如果已分配分组，也会直接显示当前分组
- 比赛列表页顶部现在也会额外提供一条统一的“当前下一步”提示：
  - 游客会优先看到“先登录或注册，随后回到原比赛”
  - 已登录但没队伍时，会优先提示“先去队伍页准备队伍”
  - 已有队伍但还没报名时，会直接提示打开一场可报名比赛
  - 已经报名过某场比赛时，会优先提示回到那场比赛继续处理题目、排行榜或 Writeup
- 如果当前没有任何公开比赛：
  - 管理员会直接看到“去管理端建赛”的空状态入口
  - 普通用户会直接看到“当前还没有公开比赛”的解释，而不是只剩一行“暂无比赛”
- 当访客还未登录时：
  - 比赛列表卡片会同时给出“登录”和“注册”两个入口
  - 比赛详情页里的“当前行动”区域也会保留这两个入口
  - 两个入口都会继续携带当前比赛 `redirect`，方便完成登录或注册后回到原比赛
- 管理员在比赛详情页概览侧边可直接跳转：
  - `/console`
  - `/console/admin`
- 管理端比赛页可以直接处理参赛队伍：
  - 修改 `pending / accepted / rejected`
  - 移除错误报名
- 队伍页统一改为通过 `$api` 调用后端，直接复用 cookie 会话
- `$api` 已支持 OpenAPI 风格路径参数，例如 `/api/games/{id}`

## 当前约束

- 未登录访客只能浏览公开比赛的基础元信息，不能直接报名、提交 Flag 或读取完整题面
- 只有 `accepted` 状态的队伍可以在比赛进行中提交 Flag
- 只有 `accepted` 状态的队伍可以提交要求中的比赛 Writeup
- 如果比赛设置了 `writeup_deadline`，超过截止时间后将不能再更新 Writeup
- `pending` / `rejected` 队伍不会获得正式参赛资格
- 为避免提前泄题，未满足参赛条件的用户只会拿到题目标题、分类、分值和解题统计
- `pending` / `rejected` 报名可以主动撤回
- `accepted` 报名一旦通过就不会再开放撤回，队伍视为锁定
- 队伍只要仍处于某场“未结束且已通过报名”的比赛里，成员结构就会同步锁定：
  - 不能再通过邀请码加入新成员
  - 普通成员不能退队
  - 队长不能移除成员
- 当对应比赛已经结束后，这个锁定会自动解除，队伍可以继续为下一场比赛调整成员
- `auto_accept` 模式只跳过审核，不会绕过比赛开始/结束时间限制
- 比赛必须处于 `active` 才会开放报名
- 比赛只有在 `active` 且到达开始时间后才允许提交 Flag
- 即使队伍已经 `accepted`，在比赛开始前也不会下发完整题面内容
- 比赛结束后仍可查看题目和榜单，但不能再报名
- 如果比赛未开启 `practice_mode`，结束后也不能再继续提交 Flag
- 如果比赛开启了 `practice_mode`，已通过报名的队伍可以在赛后继续补题提交
- 赛后练习提交不会影响正式排行榜分数、解题时间和一血/二血/三血统计
- 即使管理员忘记手动改状态，只要已经超过结束时间，读接口也会自动把比赛视为 `ended`
- 队伍人数上限只在比赛报名时生效，不影响队伍平时的创建、加入和成员调整
- 如果比赛启用了分组，公开榜单可以按分组查看；未启用时则只展示总榜
- 当前最小实例租约也遵循比赛状态限制：
  - 只有 `accepted` 队伍可以读取和启动动态题实例
  - 比赛开始前不能启动
  - 比赛结束后，只有开启 `practice_mode` 的比赛还能继续续期或重启实例

## 最小实例接口

- `GET /api/games/{id}/challenges/{challengeId}/instance`
  - 返回当前队伍在这道动态题上的实例状态
  - 可能是 `idle` 或 `running`
- `POST /api/games/{id}/challenges/{challengeId}/instance`
  - 首次调用会创建一条最小租约
  - 已存在租约时会把它续期
- 当前响应会带回：
  - `status`
  - `provider`
  - `image`
  - `launch_url`
  - `expires_at`
  - `seconds_left`
  - `can_start`
  - `can_renew`
  - `policy`
    - `lease_duration_minutes`
    - `extension_duration_minutes`
    - `renewal_window_minutes`
    - `team_active_limit`
- 这两个接口现在也已经补进 OpenAPI 契约
  - 前端比赛页已改为走统一 `$api` 类型化调用
  - 后续如果替换成真实容器 provider，前后端字段可以继续沿用这一层响应结构

## Provider 抽象

- 当前后端已经把“实例续租/启动”逻辑从比赛服务里抽成了一个最小 provider 接口
- 当前后端已经把“实例销毁”也纳入同一套 provider 生命周期
  - destroy 会先调用 provider 清理钩子
  - 只有 provider 清理成功后，才会删除数据库里的本地租约记录
- 当前单体后端还会定期扫描已过期租约
  - 过期后会优先尝试调用 provider 清理
  - 清理成功后会删除本地租约记录
  - 这样即使选手不手动点击“销毁实例”，当前本地实现 / 后续真实 provider 也能逐步对齐平台回收流程
- 现在默认仍然是基础实现：
  - 只负责生成租约状态
  - 销毁时也只会走空实现，不会真的回收 Docker / K8s 容器
  - 不会真的起 Docker / K8s 容器
- 但如果后端显式开启了 `INSTANCE_DOCKER_PROVIDER_ENABLED=true`：
  - `"docker"` provider 会被替换成一个本地 Docker CLI provider
  - 会真实执行 `docker run / inspect / rm -f`
  - 会根据 `runtime.expose` 回填实际 `host / port / launch_url`
- 这个基础 provider 现在也支持一层模板展开：
  - `{{game_id}}`
  - `{{challenge_id}}`
  - `{{team_id}}`
  - `{{user_id}}`
  - `{{team_hash}}`
- 适合先把动态题的连接入口做成“每队不同但稳定”的本地形态
  - 例如 `connection.url` 可以写成 `/local-instance/{{game_id}}/{{challenge_id}}/{{team_hash}}?team={{team_id}}`
  - 当前会把这类地址作为实例详情页入口使用；如果未启用真实 provider，仍然不会真的创建容器或反代
- 比赛页现在会把这类动态题显示成更明确的“实例面板”：
  - 会区分入口模板、当前队伍实例、实例详情入口
  - 会显示剩余租约时间、续期时间和 15 秒自动同步状态
  - 已经支持在选手侧手动销毁当前队伍实例，便于重置当前租约
  - 续期现在采用窗口控制：只有在到期前 10 分钟内才开放
  - 当前还会显示“每支队伍同时运行实例上限”，让本地检查也先遵守最小资源约束
- 这样做是为了先稳定实例生命周期与资源约束模型
  - 后续可以按 `runtime.provider` 继续接 `docker`、`k8s` 或代理型 provider
  - 当前 `docker`、`k8s`、`proxy` 等名称都会先落到同一套基础 provider
  - 当前最小租约也已经补上“每队运行中实例数上限”，避免本地环境无限制地启动动态题实例

## 题目内容格式

- 当前管理员通过题目表单直接维护：
  - `description`：题面正文
  - `hints`：JSON 字符串数组
  - `attachments`：JSON 字符串数组
- 比赛页会自动解析 `hints` 和 `attachments` 并展示为可读列表

## 计分说明

- 当前平台已统一普通题目提交和比赛内题目提交的动态计分逻辑
- 前三解仍保留 `first / second / third` 血标记
- 当前血标记只作为展示元数据，不额外乘分
- 动态分数按 `min_score + (base_score - min_score) * e^(-decay_rate * solves_before)` 计算
