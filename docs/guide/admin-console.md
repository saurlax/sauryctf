# 管理端

当前管理端先实现了一个最小闭环页面：

- 路径：`/console/admin`
- 仅 `admin` / `super_admin` 角色可见
- 非管理员访问时会直接跳回 `/console`
- 能力：
  - 创建比赛
  - 修改比赛名称、描述、公告、时间
  - 配置比赛分组
  - 修改比赛状态、报名模式与公开性
  - 删除比赛
  - 导入比赛
  - 导出比赛
  - 审核参赛队伍状态（待审核 / 已通过 / 已拒绝）
  - 移除错误报名
  - 创建题目
  - 修改题目基础信息、提示、附件链接与计分参数
  - 从管理端删除题目
  - 将已有题目挂载到比赛
  - 查看比赛参赛队伍与当前得分
  - 查看并审核比赛 Writeup
  - 查看某场比赛当前已挂载题目，以及解题数与一血 / 二血 / 三血
  - 将题目从比赛中移除

## 交互约定

- 创建比赛、创建题目、比赛挂题、编辑比赛、编辑题目、导入比赛包、发布公告这类录入或低频维护动作统一收进 `UModal`
- 删除、销毁、移除报名等高风险动作统一使用确认弹层，不再直接调用浏览器原生确认框
- 页面主体优先保留概览、设置、审核和监控；避免把与 `UModal` 重复的页内大表单长期留在正文里
- 比赛录入、题目录入、导入比赛包、发布公告这类入口区也应尽量按“维护入口面板”统一组织，避免为单个低频动作单独保留一整张空卡
- `UModal` 在页面模板底部集中声明属于正常实现，关键约束是低频录入和确认动作不要在正文区重复展开

## 当前设计原则

- 先复用现有后端 admin API
- 前端保持极简，只做表单输入和结果反馈
- 管理端顶部现在会提供一组“配置步骤概览”：
  - 先创建比赛
  - 再创建题目
  - 再挂题
  - 最后切到 `active` 并检查公开页
  - 步骤按钮现在会尽量自动带上当前最合适的比赛上下文，并滚动到对应表单区域
- 创建比赛与创建题目表单现在都提供快速填充按钮：
  - 比赛表单会预填一份公开比赛默认值
  - 题目表单会预填一份静态题默认值
- 录入入口当前进一步统一为两类面板：
  - 比赛录入：负责创建、编辑、导入比赛
  - 题目维护：负责创建、编辑、模板填充和跳转挂题
- 这两块入口文案继续保持后台化表达：
  - 使用“维护”“录入约定”“当前上下文”等术语
  - 避免测试式提示、初始化式说明和过强的步骤引导口吻
- 管理端顶部现在还提供三类预置方案入口：
  - 公开比赛：自动创建一场公开比赛和一道静态题目，并自动完成挂题
  - 动态实例比赛：自动创建一场公开比赛和一道队伍独立实例题
  - 容器实例比赛：自动创建一场公开比赛和一道容器 Web 题
  - 创建成功后会自动切换到这场比赛的管理上下文，便于继续配置比赛设置、挂题和公开展示
  - 这些入口用于建立正式比赛配置的起点，不应被当作最终可发布配置直接上线
  - 自动创建出的比赛名、题目名和 Flag 现在会使用平台基础命名，不再写入 `replace_me` 这类明显脚手架值
- 顶部入口区现在进一步合并为单张“管理入口”卡：
  - 左侧集中放比赛创建、预置方案和表单填充入口
  - 右侧直接展示当前维护顺序，减少顶部重复分段
- 管理端可见文案当前继续保持后台化表达：
  - 入口区统一使用“预置方案”“维护”“检查”“概览”等表述
  - 避免“快速起步”“继续配置”这类偏引导式口吻
- 即使后端仍保留历史接口路径（例如 `/api/auth/setup-status`）：
  - 控制台页面内部状态名也应优先使用中性表达，如 `authStatus`、`passwordSecurityRisk`
  - 不把 `setup`、初始化或向导式语义继续扩散到页面实现和计算属性命名里
- 当当前比赛使用容器实例模板时，管理端会额外显示一张容器实例检查清单：
  - 当前优先根据题目 `container_spec.metadata.purpose` 识别模板类型，不再依赖题目标题必须保持固定值
  - 先提醒管理员用普通用户报名比赛
  - 再提醒去公开页启动实例
  - 然后回到管理端检查实例监控里的 `host:port / launch_url` 回填
  - 最后确认销毁回收是否回到 `idle`
- 管理端现在还会提供“当前比赛视图”：
  - 左侧直接汇总当前状态、公开性、挂题数、报名队伍数和 Writeup 数
  - 右侧保留最关键的发布前检查项，减少中段重复卡片
  - 每条检查项现在也带有快捷动作，可直接跳到比赛设置、挂题区、编辑表单或公开页
  - 如果当前已经有比赛，但你还没有选中上下文：
    - 空状态会直接给出一场可继续配置的比赛
    - 并根据当前情况提示“先去挂题”或“先去比赛设置”
- 同步加载比赛、题目列表，方便管理时直接选择
- 选中比赛后直接查看当前挂题，减少管理切换成本
- 当前挂题列表使用独立的 admin 查询接口，不复用选手题目可见性逻辑
- 题目表单现在额外支持 `container_spec`
  - 使用 JSON 记录实例 URL、host/port、连接命令、代理入口等“接入信息”
  - 当前版本已经可以承接动态实例链路，也保留了继续扩展不同 provider 的空间
  - 如果题目类型是 `dynamic`，管理端保存前会先检查 `container_spec` 是否是有效 JSON，以及 `runtime.provider / runtime.image / runtime.expose` 是否完整
  - 如果仍保留模板镜像名或模板主机地址，管理端会先阻止保存，避免把明显未完成的动态题配置直接发到比赛里
  - “实例预览”会直接标注当前接入模式，例如固定接入入口、平台回填入口或每队独立入口，方便在保存前快速确认当前配置方向
- 创建题目时现在额外提供三类快速模板：
  - `Web 实例`：用于统一入口的 Web 题
  - `Pwn 服务`：用于 `host / port / nc` 型服务题
  - `容器 Web`：默认使用 `nginx:alpine + expose: [80]`
    - 会额外预填运行提示、预期服务说明，以及“实例入口会由平台回填”的说明
- 题目表单现在还额外提供一个 `队伍独立入口` 方案：
  - 用于创建带 `runtime.provider` / `runtime.image` 的动态题
  - 会把 `connection.url`、`host`、`port`、`command` 预填成带占位符的配置
  - 默认模板里的入口地址和镜像名只是结构示例，管理员应在上线前替换为真实分发地址和可用镜像
  - 当前支持的占位符：
    - `{{game_id}}`
    - `{{challenge_id}}`
    - `{{team_id}}`
    - `{{user_id}}`
    - `{{team_hash}}`
  - 用于先确定“每队不同入口”的数据结构，再接入真实实例 provider
- 预填内容只负责提供字段结构和默认写法：
  - Flag、访问入口、镜像、附件和接入说明都应由管理员按正式比赛配置补全
  - 其中的示例值不应直接用于上线比赛
- 如果你已经开启 `INSTANCE_DOCKER_PROVIDER_ENABLED=true`：
  - 可先用“容器 Web”方案配置真实容器实例
  - 如果只想检查入口解析与租约状态，再用“队伍独立入口”方案
- 即使题目处于隐藏状态，管理员仍能在挂题视图里看到完整题面、提示和附件字段，便于核对配置
- 管理端当前使用到的删除比赛、导入/导出比赛、挂题详情、Writeup 审核接口都已补进 OpenAPI，前端统一通过同一套类型化 API 调用
- 管理端里的公告、报名、Writeup、提交、实例和资源列表等高频无数据区域已统一为正式空态：
  - 会说明当前为什么为空
  - 会直接给出对应入口，例如创建比赛、创建题目、去公开页或跳转到对应管理区块
- 选手提交 Writeup 后，管理员列表现在会立即读到最新记录，不需要额外刷新数据库或重启服务
- 比赛基础信息与比赛状态拆成两个独立表单，避免误操作
- 比赛基础信息中单独加入“公告”字段，便于在比赛详情页面向选手展示补充规则或提醒
- 这块现在在页面里统一改叫“规则补充”：
  - 用于填写长期有效的规则说明
  - 赛时通知仍通过公告区单独发布
- 比赛状态编辑先只开放 `draft / active / ended`、`registration_mode` 与 `is_public`
- 报名模式当前支持：
  - `review`：提交报名后进入待审核
  - `auto_accept`：提交报名后直接获得参赛资格
- 比赛扩展配置当前支持：
  - `practice_mode`：标记比赛结束后是否继续保留练习入口
  - `writeup_required`：标记比赛是否要求赛后提交 Writeup
  - `writeup_deadline`：留空即可；如果填写，不能早于比赛 `end_time`
- 比赛时间字段当前会在前端先做一层关系校验：
  - `end_time` 必须晚于 `start_time`
  - `scoreboard_freeze_at` 不能早于开始时间，也不能晚于结束时间
  - `writeup_deadline` 不能早于比赛结束时间
  - 如果开启 `writeup_required`，则必须同时填写 `writeup_deadline`
  - 比赛切换到 `active` 前必须至少挂载一道题目
  - 已处于 `active` 的比赛不能把挂题删到 0 道
- 比赛分组当前支持：
  - 在创建比赛、编辑比赛、比赛设置里直接维护分组列表
  - 使用换行或逗号分隔，例如：`本科组, 公开组`
  - 管理员可以在参赛队伍卡片里给每支队伍分配一个已配置分组
  - 分组会同步影响公开排行榜筛选
- 队伍人数上限当前支持：
  - `0`：不限制人数
  - 正整数：超过人数上限的队伍无法报名该比赛
- 封榜时间当前支持：
  - 留空：公开榜单不封榜
  - 指定时间：到达该时间后公开榜单冻结，但比赛仍可继续提交
- 创建比赛、编辑比赛和比赛设置表单现在都会直接显示一段规则摘要：
  - 方便在保存前核对报名方式、邀请码、人数限制、分组、练习模式、Writeup 和封榜语义
- 参赛队伍卡片允许直接修改报名状态，便于管理员放行或拒绝报名申请
- 参赛队伍卡片会同时展示并编辑：
  - 当前报名状态
  - 当前所属分组
  - 当前得分和解题数
- 题目表单支持直接录入：
  - 题目描述
  - 提示列表（JSON 数组）
  - 附件链接（JSON 数组）
- 题目附件现在也支持直接上传本地文件：
  - 管理端会把返回的 `/attachments/**` 路径自动写入当前附件 JSON 数组
  - 外部链接仍然可以继续手工补充
- 题目表单当前也支持配置“错误尝试上限”：
  - `0` 表示不限制
  - 达到上限后，当前队伍在该比赛里不能再继续提交这道题
- 题目编辑先覆盖标题、描述、分类、难度、可见性与基础计分参数
- 题目删除放在已加载资源列表中，并在前端做二次确认
- 资源列表提供快捷按钮，可一键填充当前管理表单
- 资源列表现在也提供一层轻量搜索筛选：
  - 比赛列表支持按关键字和比赛状态筛选
  - 题目列表支持按关键字、分类和可见性筛选
  - 适用于比赛与题目数量逐渐增多后的日常定位与维护
- 参赛队伍列表先展示报名时间、状态、当前得分和解题数
- 删除比赛只清理当前比赛范围内的数据：
  - 报名记录
  - 比赛内解题记录
  - 比赛 Writeup 记录
  - 比赛挂题关系
  - 题库中的原始题目不会被一起删除，方便后续复用
- 导出比赛会下载一个 ZIP 包：
  - 当前包内只有 `game.json`
  - 包含比赛基础信息与已挂载题目的完整配置
  - 当前导出版本为 `sauryctf.export.v2`
  - 也会包含 `practice_mode`、`writeup_required`、`writeup_deadline`
  - 外部附件链接会继续保留为 JSON 数组
  - 如果题目附件使用本地 `/attachments/**` 路径，这些文件也会一起打包进 ZIP
- 导出榜单会下载一个单独的 ZIP 包：
  - 当前包内包含 `scoreboard.json`
  - 也会额外包含 `rankings.csv` 和 `challenge-stats.csv`
  - 可用于赛后归档、手工复核排名或继续导入表格工具做复盘
  - 如果管理页当前切到了某个分组，也可以直接导出这个分组口径的榜单包
- 导出 Writeup 会下载一个单独的 ZIP 包：
  - 当前包内包含 `writeups.json`
  - 也会额外包含 `writeups.csv`
  - 每支队伍的正文还会拆成单独的 `writeups/team-*.md`
  - 可用于赛后批量归档、转交评审或继续做线下复盘整理
- 导出提交记录会下载一个单独的 ZIP 包：
  - 当前包内包含 `submissions.json`
  - 也会额外包含 `submissions.csv`
  - 当前记录来源于数据库里真实持久化的比赛提交日志
  - 会同时包含正确提交、错误 Flag、重复提交，以及练习模式下的提交
  - 每条记录现在也会保留选手原始提交内容 `submitted_flag`
  - 可用于赛后归档、核对解题时间线或继续导入表格工具做复盘
- 管理端现在也会提供“最近提交”只读卡片：
  - 支持按结果类型筛选：
    - 全部结果
    - 只看正确提交
    - 只看错误 Flag
    - 只看重复提交
  - 支持按条数切换最近 `20 / 50 / 100 / 200` 条
  - 便于管理员在不解压归档的情况下直接观察赛时提交节奏
- 管理端现在也会提供“可疑提交线索”只读卡片：
  - 当前规则只做轻量排查，不做自动判定
  - 线索口径是：同一场比赛里，多支队伍向同一道题提交了相同的错误 Flag
  - 当前不会分析练习模式下的提交
  - 可帮助管理员快速发现可能的错误 Flag 传播或队伍间串通线索
- 管理端现在也会提供“实例监控”只读区：
  - 管理员现在也可以直接手动销毁某条实例租约
  - 可用于处理：
    - 卡住但还没自动回收的实例
    - 已经过期但还想立即清理的本地实例 / provider 租约
    - 需要强制为某支队伍重置动态题环境的情况
  - 当前入口位于 `/console/admin` 的“赛事监控”总览里
  - 只要选中比赛，就能直接看到这场比赛当前的动态实例租约
  - 当前会展示：
    - 队伍名和题目名
    - provider / image
    - 如果 provider 已经回填了真实连接信息，也会显示当前 `host:port`
    - 启动时间、最近续期时间、到期时间
    - 剩余秒数或是否已过期
    - 当前解析后的实例入口 `launch_url`
  - 对于真实本地 Docker provider：
    - 会额外用更明确的“本地 Docker”标识展示
    - 管理员可以直接复制入口，或一键打开当前实例地址
  - 可用于赛时快速判断：
    - 是否已经有队伍成功拉起实例
    - 是否存在大量过期租约
    - 某个动态题当前是否仍在被使用
- “赛事监控”总览当前进一步压缩为三层信息：
  - 待处理事项：集中展示待审核报名与待审 Writeup
  - 最新公告：保留最近对外通知入口
  - 实例监控：保留运行中 / 已过期实例摘要与明细
- 低频维护动作当前还集中进一张“维护入口”卡：
  - 导入比赛包
  - 发布公告
  - 打开公开页
- 运营区当前也进一步统一了页头结构：
  - 参赛队伍：保留队伍总数、待审核数、已通过数
  - Writeup 审核：保留总数、待审数和导出入口
  - 比赛公告：保留公告总数与发布入口
- 导入比赛当前支持上传管理端导出的 ZIP 包：
  - 接受 `sauryctf.export.v1` 和 `sauryctf.export.v2`
  - 必须包含 `game.json`
  - 导入后会新建一场 `draft` 比赛
  - 导入成功后，管理页会自动切到这场新比赛，方便继续核对挂题、参赛队伍和 Writeup 区域
  - 导入完成后，题目内容应继续从管理端挂题视图核对，而不是直接从公开比赛题目页读取
  - 比赛的练习模式和 Writeup 配置会一起恢复
  - 题目会作为新的题库记录重新创建，再挂载到新比赛
  - `v2` 包里嵌入的本地附件会恢复到服务端 `./attachments`
  - 外部附件链接仍只按 URL 保存，不会在导入时主动下载
- 如果删除的是当前正在管理的比赛：
  - 挂题列表、参赛队伍和 Writeup 区域会一起清空，避免页面继续显示旧上下文
- 赛事监控的“运维”标签现在还会提供一个统一的“赛后归档”区：
  - 导出比赛包
  - 导出榜单包
  - 导出 Writeup
  - 导出提交记录
  - 管理员不需要再分别去各个区域找导出按钮
- 不在这一版里加入复杂的编辑、删除、批量操作
