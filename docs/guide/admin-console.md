# 管理端

当前管理端先实现了一个最小闭环页面：

- 路径：`/console/admin`
- 仅 `admin` / `super_admin` 角色可见
- 非管理员访问时会直接跳回 `/console`
- 能力：
  - 创建比赛
  - 修改比赛名称、描述、公告、时间
  - 配置比赛分组
  - 修改比赛状态、报名模式与公开性
  - 删除比赛
  - 导入比赛
  - 导出比赛
  - 审核参赛队伍状态（待审核 / 已通过 / 已拒绝）
  - 移除错误报名
  - 创建题目
  - 修改题目基础信息、提示、附件链接与计分参数
  - 从管理端删除题目
  - 将已有题目挂载到比赛
  - 查看比赛参赛队伍与当前得分
  - 查看并审核比赛 Writeup
  - 查看某场比赛当前已挂载题目，以及解题数与一血 / 二血 / 三血
  - 将题目从比赛中移除

## 当前设计原则

- 先复用现有后端 admin API
- 前端保持极简，只做表单输入和结果反馈
- 管理端顶部现在会提供一条“首次建赛路径”：
  - 先创建比赛
  - 再创建题目
  - 再挂题
  - 最后切到 `active` 并验证公开页
  - 路径按钮现在会尽量自动带上当前最合适的比赛上下文，并滚动到对应表单区域
- 创建比赛与创建题目表单现在都提供“本地验证模板”按钮：
  - 比赛模板会预填一个适合本地验证的公开比赛配置
  - 题目模板会预填一个可直接验证的 `flag{warmup}` 题目
- 管理端顶部现在还提供“一键创建验证比赛”：
  - 会自动创建一场公开比赛
  - 会自动创建一道 `flag{warmup}` 题目
  - 会自动把这道题挂到新比赛上
  - 创建成功后会自动切换到这场比赛的管理上下文，便于立刻继续检查公开页和挂题区域
- 管理端顶部现在还额外提供“一键创建动态实例验证赛”：
  - 会自动创建一场公开比赛
  - 会自动创建一道 `dynamic` 题目
  - 题目默认带 `runtime.provider = docker`，但当前这条快捷入口仍以本地实例访问页链路为主
  - `connection.url` / `host` / `command` 默认会使用 `{{team_hash}}`、`{{team_id}}` 这类占位符
  - 当前默认会落到前端本地的 `/local-instance/...` 页面，方便直接验证入口是否可访问
  - 适合直接验证 team-scoped 实例租约、模板入口解析，以及选手页里的动态题入口展示
- 管理端顶部现在还额外提供“一键创建本地 Docker 验证赛”：
  - 会自动创建一场公开比赛
  - 会自动创建一道 `Local Docker Web Lease` 动态题
  - 题目默认带 `runtime.provider = docker`、`runtime.image = nginx:alpine`、`runtime.expose = [80]`
  - 更适合直接验证真实本地 Docker provider 下的实例启动、入口回填和销毁回收
  - 这条链路仍然要求本机 Docker daemon 可用
- 当当前比赛命中了这条本地 Docker 验证题模板时，管理端还会额外显示一张“本地 Docker 验证清单”：
  - 先提醒管理员用普通用户报名比赛
  - 再提醒去公开页启动实例
  - 然后回到管理端检查实例监控里的 `host:port / launch_url` 回填
  - 最后再验证销毁回收是否回到 `idle`
- 管理端现在还会提供“当前比赛概览”和“赛前检查”：
  - 只要选中一场比赛，就会直接汇总当前状态、公开性、挂题数、报名队伍数和 Writeup 数
  - 赛前检查会优先提醒是否已经挂题、是否已有报名队伍、当前状态是否适合继续核对或直接开赛
  - 每条检查项现在也带有快捷动作，可直接跳到比赛设置、挂题区、编辑表单或公开页
  - 如果当前已经有比赛，但你还没有选中上下文：
    - “当前比赛概览”空状态会直接推荐一场最适合继续配置的比赛
    - 并根据当前情况提示“先去挂题”或“先去比赛设置”
- 同步加载比赛、题目列表，方便管理时直接选择
- 选中比赛后直接查看当前挂题，减少管理切换成本
- 当前挂题列表使用独立的 admin 查询接口，不复用选手题目可见性逻辑
- 题目表单现在额外支持 `container_spec`
  - 推荐用 JSON 记录实例 URL、host/port、连接命令、代理入口等“接入信息”
  - 当前版本先作为轻量展示层使用，后续可以继续对接真正的动态容器或靶机编排
- 创建题目时现在额外提供三类快速模板：
  - `Web 实例`：适合统一入口的 Web 题
  - `Pwn 服务`：适合 `host / port / nc` 型服务题
  - `本地 Docker Web`：默认使用 `nginx:alpine + expose: [80]`，适合直接验证真实本地 Docker provider
    - 会额外预填本地运行提示、预期服务说明，以及“实例入口会由平台回填”的说明
- 题目表单现在还额外提供一个 `每队独立入口` 模板：
  - 适合先创建带 `runtime.provider` / `runtime.image` 的动态题
  - 会把 `connection.url`、`host`、`port`、`command` 预填成带占位符的模板
  - 当前默认模板会指向本地 `/local-instance/...` 页面，方便先验证入口链路
  - 当前支持的占位符：
    - `{{game_id}}`
    - `{{challenge_id}}`
    - `{{team_id}}`
    - `{{user_id}}`
    - `{{team_hash}}`
  - 即使还没接真实容器 provider，也能先在选手页走通“每队不同实例入口”的体验
- 如果你已经开启 `INSTANCE_DOCKER_PROVIDER_ENABLED=true`：
  - 更推荐先用“本地 Docker Web”模板验证真实本地 Docker
  - 如果只想验证入口模板解析与租约状态，再用“每队独立入口”模板
- 即使题目处于隐藏状态，管理员仍能在挂题视图里看到完整题面、提示和附件字段，便于核对配置
- 管理端当前使用到的删除比赛、导入/导出比赛、挂题详情、Writeup 审核接口都已补进 OpenAPI，前端统一通过同一套类型化 API 调用
- 选手提交 Writeup 后，管理员列表现在会立即读到最新记录，不需要额外刷新数据库或重启服务
- 比赛基础信息与比赛状态拆成两个独立表单，避免误操作
- 比赛基础信息中单独加入“公告”字段，便于在比赛详情页面向选手展示补充规则或提醒
- 这块现在在页面里统一改叫“规则补充”：
  - 适合填写长期有效的规则说明
  - 赛时通知仍建议通过公告区单独发布
- 比赛状态编辑先只开放 `draft / active / ended`、`registration_mode` 与 `is_public`
- 报名模式当前支持：
  - `review`：提交报名后进入待审核
  - `auto_accept`：提交报名后直接获得参赛资格
- 比赛扩展配置当前支持：
  - `practice_mode`：标记比赛结束后是否继续保留练习入口
  - `writeup_required`：标记比赛是否要求赛后提交 Writeup
  - `writeup_deadline`：留空即可；如果填写，不能早于比赛 `end_time`
- 比赛时间字段当前会在前端先做一层关系校验：
  - `end_time` 必须晚于 `start_time`
  - `scoreboard_freeze_at` 不能早于开始时间，也不能晚于结束时间
  - `writeup_deadline` 不能早于比赛结束时间
  - 如果开启 `writeup_required`，则必须同时填写 `writeup_deadline`
- 比赛分组当前支持：
  - 在创建比赛、编辑比赛、比赛设置里直接维护分组列表
  - 使用换行或逗号分隔，例如：`本科组, 公开组`
  - 管理员可以在参赛队伍卡片里给每支队伍分配一个已配置分组
  - 分组会同步影响公开排行榜筛选
- 队伍人数上限当前支持：
  - `0`：不限制人数
  - 正整数：超过人数上限的队伍无法报名该比赛
- 封榜时间当前支持：
  - 留空：公开榜单不封榜
  - 指定时间：到达该时间后公开榜单冻结，但比赛仍可继续提交
- 创建比赛、编辑比赛和比赛设置表单现在都会直接显示一段规则摘要：
  - 方便在保存前快速核对报名方式、邀请码、人数限制、分组、练习模式、Writeup 和封榜语义
- 参赛队伍卡片允许直接修改报名状态，便于管理员快速放行或拒绝报名申请
- 参赛队伍卡片会同时展示并编辑：
  - 当前报名状态
  - 当前所属分组
  - 当前得分和解题数
- 题目表单支持直接录入：
  - 题目描述
  - 提示列表（JSON 数组）
  - 附件链接（JSON 数组）
- 题目编辑先覆盖标题、描述、分类、难度、可见性与基础计分参数
- 题目删除放在已加载资源列表中，并在前端做二次确认
- 资源列表提供快捷按钮，可一键填充当前管理表单
- 参赛队伍列表先展示报名时间、状态、当前得分和解题数
- 删除比赛只清理当前比赛范围内的数据：
  - 报名记录
  - 比赛内解题记录
  - 比赛 Writeup 记录
  - 比赛挂题关系
  - 题库中的原始题目不会被一起删除，方便后续复用
- 导出比赛会下载一个 ZIP 包：
  - 当前包内只有 `game.json`
  - 包含比赛基础信息与已挂载题目的完整配置
  - 当前导出版本为 `sauryctf.export.v2`
  - 也会包含 `practice_mode`、`writeup_required`、`writeup_deadline`
  - 外部附件链接会继续保留为 JSON 数组
  - 如果题目附件使用本地 `/attachments/**` 路径，这些文件也会一起打包进 ZIP
- 导出榜单会下载一个单独的 ZIP 包：
  - 当前包内包含 `scoreboard.json`
  - 也会额外包含 `rankings.csv` 和 `challenge-stats.csv`
  - 适合赛后归档、手工复核排名或继续导入表格工具做复盘
  - 如果管理页当前切到了某个分组，也可以直接导出这个分组口径的榜单包
- 导出 Writeup 会下载一个单独的 ZIP 包：
  - 当前包内包含 `writeups.json`
  - 也会额外包含 `writeups.csv`
  - 每支队伍的正文还会拆成单独的 `writeups/team-*.md`
  - 适合赛后批量归档、转交评审或继续做线下复盘整理
- 导出提交记录会下载一个单独的 ZIP 包：
  - 当前包内包含 `submissions.json`
  - 也会额外包含 `submissions.csv`
  - 当前记录来源于数据库里真实持久化的比赛提交日志
  - 会同时包含正确提交、错误 Flag、重复提交，以及练习模式下的提交
  - 每条记录现在也会保留选手原始提交内容 `submitted_flag`
  - 适合赛后归档、核对解题时间线或继续导入表格工具做复盘
- 管理端现在也会提供“最近提交”只读卡片：
  - 支持按结果类型筛选：
    - 全部结果
    - 只看正确提交
    - 只看错误 Flag
    - 只看重复提交
  - 支持按条数切换最近 `20 / 50 / 100 / 200` 条
  - 便于管理员在不解压归档的情况下直接观察赛时提交节奏
- 管理端现在也会提供“可疑提交线索”只读卡片：
  - 当前规则只做轻量排查，不做自动判定
  - 线索口径是：同一场比赛里，多支队伍向同一道题提交了相同的错误 Flag
  - 当前不会分析练习模式下的提交
  - 适合管理员快速发现可能的错误 Flag 传播或队伍间串通线索
- 管理端现在也会提供“实例监控”只读区：
  - 管理员现在也可以直接手动销毁某条实例租约
  - 适合处理：
    - 卡住但还没自动回收的实例
    - 已经过期但还想立即清理的本地实例 / provider 租约
    - 需要强制为某支队伍重置动态题环境的情况
  - 当前入口位于 `/console/admin` 的“赛事监控”总览里
  - 只要选中比赛，就能直接看到这场比赛当前的动态实例租约
  - 当前会展示：
    - 队伍名和题目名
    - provider / image
    - 如果 provider 已经回填了真实连接信息，也会显示当前 `host:port`
    - 启动时间、最近续期时间、到期时间
    - 剩余秒数或是否已过期
    - 当前解析后的实例入口 `launch_url`
  - 对于真实本地 Docker provider：
    - 会额外用更明确的“本地 Docker”标识展示
    - 管理员可以直接复制入口，或一键打开当前实例地址
  - 适合赛时快速判断：
    - 是否已经有队伍成功拉起实例
    - 是否存在大量过期租约
    - 某个动态题当前是否仍在被使用
- 导入比赛当前支持上传管理端导出的 ZIP 包：
  - 接受 `sauryctf.export.v1` 和 `sauryctf.export.v2`
  - 必须包含 `game.json`
  - 导入后会新建一场 `draft` 比赛
  - 导入成功后，管理页会自动切到这场新比赛，方便继续核对挂题、参赛队伍和 Writeup 区域
  - 导入完成后，题目内容应继续从管理端挂题视图核对，而不是直接从公开比赛题目页读取
  - 比赛的练习模式和 Writeup 配置会一起恢复
  - 题目会作为新的题库记录重新创建，再挂载到新比赛
  - `v2` 包里嵌入的本地附件会恢复到服务端 `./attachments`
  - 外部附件链接仍只按 URL 保存，不会在导入时主动下载
- 如果删除的是当前正在管理的比赛：
  - 挂题列表、参赛队伍和 Writeup 区域会一起清空，避免页面继续显示旧上下文
- 赛事监控的“运维”标签现在还会提供一个统一的“赛后归档”区：
  - 导出比赛包
  - 导出榜单包
  - 导出 Writeup
  - 导出提交记录
  - 管理员不需要再分别去各个区域找导出按钮
- 不在这一版里加入复杂的编辑、删除、批量操作
