# 登录与注册

当前前端提供独立的 `/login` 与 `/register` 页面，右上角导航固定保留两个独立入口：

- `登录`
- `注册`
- 不合并为单一“登录/注册”按钮
- 不提供单独的初始化页或首次启动引导页

## 默认管理员

只有在 `users` 表完全为空时，后端启动才会自动创建默认管理员：

- 用户名：`admin`
- 密码：`sauryctf`

一旦数据库里已经有任意用户：

- 后端不会再补建 `admin`
- 首页和 `/login` 也不会继续把默认口令当成常规入口提示
- `/api/auth/setup-status` 会立即切到 `bootstrap_admin_available = false`
- `/api/auth/setup-status` 不会向访客回传默认管理员口令
- `/api/auth/setup-status` 现在只保留“空库首次启动是否仍可使用默认管理员”这一类访客态信号

需要注意：

- `admin / sauryctf` 不是“系统保底账号”，而是“仅空库首次启动可用的初始账号”
- 只要库里已经存在任意用户，就不应再因为缺少 `admin` 而自动补回该账号

## 管理员改密

- 当前已新增 `/console/account`
- 登录后的任意账号都可以在这里修改自己的登录密码
- 接口为 `POST /api/auth/change-password`
- 修改密码成功后，当前账号的既有会话会立即失效，需要使用新密码重新登录
- 如果当前登录的是仍在使用初始密码的管理员账号：
  - `/api/auth/security-status` 会返回 `password_change_recommended = true`
  - `/console` 会显示一条显式安全提醒
  - `/console/account` 会继续高亮“立即改密”的风险提示
- `/api/auth/security-status` 当前只承担登录后安全状态信号：
  - 当前登录用户是否仍在使用初始管理员密码
- `/console/account` 当前采用更紧凑的状态卡：
  - 顶部仅保留一条账号安全状态提示
  - 右侧改为账号信息与安全状态摘要，不再堆叠长段建议文案
  - 改密表单通过弹层触发，不直接铺在页面正文中

建议顺序：

1. 空库首次启动后先用 `admin / sauryctf` 登录
2. 进入 `/console/account` 修改默认管理员密码
3. 使用新密码重新登录
4. 再创建比赛、题目和普通选手账号

## 普通用户链路

- 直接访问 `/register` 注册成功后，会自动登录并直接进入 `/console/team`
- 队伍页会继续按是否已有队伍、是否来自比赛页展示当前状态
- `/register` 当前保持为精简表单页：
  - 表单页底部只保留注册后去向提示与常用入口
  - 不额外展示流程化说明卡片
- 如果是从比赛详情页跳到 `/register?redirect=/games/{id}`：
  - 注册成功后会先进入 `/console/team?redirect=/games/{id}`
  - 队伍页会明确提示当前来自比赛页
  - 创建队伍或加入队伍成功后，会自动回到原比赛

## 登录链路

- `/login` 支持用户名或邮箱登录
- 如果当前链接带有 `redirect` 查询参数，登录成功后会优先跳回该地址
- 如果没有 `redirect`，统一默认进入 `/console`
- `/login` 当前保持为精简表单页：
  - 页面仅展示登录表单、跳转目标摘要和常用入口
  - 提交中会锁定输入并显示 loading，避免重复提交
  - 页脚仅保留一个轻量的注册提示和常用入口按钮
  - 不承载额外的初始化提示或引导流程
- 首页、`/login` 与 `/register` 的公开文案应保持中性：
  - 不在页面中直接展示默认管理员口令
  - 不把公开认证入口写成初始化向导或演示流程
- `/register` 当前也保持同样的轻量结构：
  - 页面仅展示注册表单、注册后去向摘要和返回登录提示
  - 用户名长度约束与后端保持一致，当前至少为 3 个字符
  - 提交中会锁定输入并显示 loading，避免重复提交
  - 不额外叠加流程说明卡片
- `/console/account` 当前继续保持正式状态表达：
  - 使用“建议优先更新登录密码”“账号可继续使用”这类直接标题
  - 避免“当前状态：...”式提示语
  - 低频敏感操作通过弹层完成，页面主体优先保留状态信息与摘要
- `/console/team` 在未组队状态下也继续保持轻量：
  - 只保留比赛跳转提示、入口说明和一条统一的队伍说明
  - 创建队伍与加入队伍统一通过弹层完成
  - 不再为相同流程重复堆叠多块说明卡
- `/console/team` 在已组队状态下也继续收敛为状态页：
  - 右侧改为一张统一的“队伍管理摘要”或“当前操作范围”卡
  - 使用摘要行和一条建议提示统一呈现角色、锁定状态、邀请码使用范围和退队限制
- Nuxt 开发环境下，认证相关首屏请求现在区分了两条访问路径：
  - 浏览器侧继续走同源 `/api/**`，保留 Cookie 与 dev proxy 行为
  - Nuxt 服务端渲染时改为直接访问 `http://127.0.0.1:8080/api`
  - 这样像首页、控制台这类会在首屏拉取认证状态的页面，就不会再把 `/api/**` 误判成 Nuxt 自己的内部路由

## 推荐验证顺序

1. 空库启动后先用 `admin / sauryctf` 登录
2. 创建一场公开比赛并激活
3. 再注册普通账号
4. 创建或加入队伍
5. 回到比赛详情继续报名、提 Flag、看榜
