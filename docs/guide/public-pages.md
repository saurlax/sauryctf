# 公开页

公开页当前以“状态可读、操作克制”为主，避免做成流程引导页。

## 比赛详情页

- 比赛详情页的 Writeup 状态优先复用后端返回的参与状态字段
- 对已登录但尚未加入队伍的用户：
  - 页面只展示比赛与报名状态
  - 不应直接请求队伍级 Writeup 数据，避免把“未组队”误判成页面加载失败
- 如果后端已经返回 `writeup_deadline_passed`：
  - 前端应优先使用该字段展示截止状态
  - 不再单纯依赖页面本地时间自行推断
- 比赛列表页与比赛详情页应共享同一套比赛阶段判断：
  - `draft`
  - `before_start`
  - `active`
  - `ended`
- 不要只根据 `game.status` 把“已激活但尚未到开始时间”的比赛当成进行中
- 比赛详情页内部的动作判断也应尽量直接依赖阶段值：
  - 例如报名入口、提交权限、题目卡状态、下一步动作
  - 不要反过来依赖 `未开始 / 进行中 / 已结束` 这类展示文案做业务分支

## 文案约定

- 首页入口使用平台化命名，例如：
  - `管理入口`
  - `选手账号`
  - `公开比赛`
- 比赛详情页优先展示状态摘要，而不是编号步骤：
  - `账号状态`
  - `队伍状态`
  - `报名状态`
  - `提交权限`
- 参赛状态区使用摘要行配合单条说明：
  - 摘要行负责说明当前事实
  - 单条说明负责补充当前限制或下一步
- 比赛详情页顶部当前保持：
  - 一条主参赛状态结论
  - 一条紧凑的下一步操作条
  - 不再为同一状态连续堆叠多块大提示框
- 状态提示标题避免使用 `当前状态：...` 这类前缀，直接使用结论式标题：
  - `需要先登录`
  - `报名待审核`
  - `比赛已结束`
- 比赛详情页里的实例、报名、Writeup 状态提示优先写成短结论：
  - 用 `登录后可查看实例状态` 代替长段解释
  - 用 `报名待审核`、`报名未通过`、`已具备参赛资格` 代替流程式描述
  - 需要额外说明时，描述行只补充开放范围或限制，不重复整段下一步教程
- 比赛详情页右侧动作区与题目卡提示优先使用“当前已满足什么 / 当前还缺什么”这类短句：
  - 避免重复解释完整报名流程
  - 避免在多个卡片里反复描述同一条下一步操作

## 交互约定

- 未登录访客仍可浏览公开比赛基础信息、题目标题与排行榜。
- 登录、注册、队伍管理等动作可以保留，但只作为明确入口，不展开成长流程说明。
- 不单独增加首次启动引导页、初始化页或 onboarding 式说明页。
- 比赛详情页右侧面板优先承担：
  - 当前行动
  - 参赛状态
  - 参赛限制
  - 队伍与分组
- 比赛详情页左侧信息区优先拆成明确的信息层次：
  - 基础信息
  - 规则摘要
  - 赛时通知
  - 避免把规则、补充信息和公告长期堆成一个连续文本块
- 比赛详情页的 Writeup 区也保持相同思路：
  - 审核状态优先使用摘要行
  - 提交要求优先使用逐条状态块
  - 避免长段说明直接堆成一整块正文
- 实例详情页也遵守相同原则：
  - 页面主体优先保留租约状态、实例入口和策略摘要
  - 启动、续期、刷新保留为直接动作
  - 销毁实例这类危险操作在提交前进入 `UModal` 确认，不直接生效
- 如果页面顶部已经给出完整的下一步结论：
  - 右侧“当前行动”区优先改为结构化摘要和快捷入口
  - 避免在右侧再次重复整块结论型提示
- 不再保留“第 1 步 / 第 2 步”式参与说明卡片。
- 空态按钮与受限状态按钮优先使用明确入口动作：
  - `登录账号`
  - `管理队伍`
  - `查看比赛概览`
- `UModal` 集中声明在页面模板底部是正常实现；关键在于低频录入、确认和危险操作应收进模态，不要把临时确认块直接留在正文里。
- 如果正文区只保留触发按钮，而弹层内容集中在模板底部维护，视为符合页面结构约定。
